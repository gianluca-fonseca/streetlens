billed ~2565 tokens. Measured against the real API, that is not what happened.

and the first cut of this fix repeated the mistake in a smaller way:

  SYSTEM_PROMPT (the cacheable prefix)   ~2711 tokens

  user instruction                         ~20 tokens
  ------------------------------------------------------
  static request, estimated              ~4628 tokens
  static request, actual (cached_tokens)  4480 tokens   <- estimate runs ~3% high

So a CORRECT call cannot come in under ~4500 tokens no matter what the image
costs. The old flat 2600 ceiling sat ~1900 tokens BELOW the floor of a correct
call: it was never a cost breaker, it was a tripwire on our own prompt. A first
attempt at this fix derived the ceiling from the prompt alone (3911) and still
fired on a correct call at 4619 — the schema is ~40% of the static request and
is billed like anything else. inputTokenCeiling() now measures prompt + schema
+ user turn, so editing either file cannot silently break the breaker.
